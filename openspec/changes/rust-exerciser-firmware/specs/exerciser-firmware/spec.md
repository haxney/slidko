# Delta: exerciser-firmware

## REMOVED Requirements

### Requirement: JSON-lines command interface
**Reason**: Replaced by a committed protobuf schema. JSON's implicit typing
admits silent cross-language ambiguity, fails open on unknown/missing fields,
and forced binary `payload` to be carried as a C string â€” corrupting any
`0x00`/`"` byte. A canonical `.proto` forbids these by construction and gives
field-numbered schema evolution. (This carries over the decision from the
superseded `exerciser-protobuf-wire` change, now realised in Rust.)
**Migration**: Firmware and host speak length-delimited protobuf per the new
"Protobuf command interface" requirement; the v1 command set and the
`id`-echo / `ok|err` contract are preserved unchanged â€” only the encoding
differs. `protoc --decode_raw` recovers human-readable wire dumps.

### Requirement: Self-provisioning pinned SDK
**Reason**: The pico-sdk (C/CMake) is retired with the C firmware; a Rust
firmware has no pico-sdk to vendor. Toolchain reproducibility is now a Cargo /
`rust-toolchain.toml` concern, covered by the new "Reproducible pinned Rust
toolchain" requirement.
**Migration**: Remove the vendored `firmware/vendor/pico-sdk` and its clone
step; pin crate versions in `Cargo.lock` and the toolchain + `thumbv8m` target
in `rust-toolchain.toml`. No network clone of an SDK is required to build.

## ADDED Requirements

### Requirement: Rust firmware platform and toolchain
The firmware SHALL be implemented in Rust for the RP2350A **ARM Cortex-M33**
core (`thumbv8m.main-none-eabi`), using `rp235x-hal`/`embassy-rp` and
`embassy-usb` for the USB CDC transport. Push-pull protocol stimulus (WS2812,
DShot150/300/600) SHALL be generated by PIO programs whose bit-timing constants
are those in docs/EXERCISER.md, expressed as inline PIO assembly (`pio-proc`);
one DShot PIO program SHALL serve all three rates via the state-machine clock
divider (T1H = 2Ă—T0H). The build SHALL flash via UF2/BOOTSEL or `picotool` and
SHALL NOT depend on account-walled, Windows-only, or interactive-debugger
(probe-rs RP2350) tooling.

#### Scenario: DShot timing constants match the spec
- **WHEN** the DShot PIO program and its clock-divider computation are unit-tested on the host for DShot150/300/600
- **THEN** the derived bit period, T0H, and T1H match the docs/EXERCISER.md table (6.67/3.33/1.67 Âµs periods; T1H = 2Ă—T0H) with no target hardware attached

#### Scenario: Open, headless toolchain
- **WHEN** the firmware is built from a clean checkout on Linux
- **THEN** the build completes with only Rust stable + the `thumbv8m.main-none-eabi` target and pinned crates, using no account-walled or Windows-only tools, and produces a UF2 image

### Requirement: Reproducible pinned Rust toolchain
The firmware build SHALL be reproducible from pinned inputs without a manual
pre-install: crate versions pinned in `Cargo.lock`, and the toolchain channel
plus the `thumbv8m.main-none-eabi` target pinned in `rust-toolchain.toml`. No
SDK or vendored C tree SHALL be committed, and no interactive step SHALL be
required to obtain build inputs.

#### Scenario: Clean checkout builds from pinned inputs
- **WHEN** the firmware is built twice from a clean checkout â€” once with an empty Cargo cache, once with it warm
- **THEN** both builds resolve the identical pinned crate versions and target, and neither a vendored SDK nor a build tree appears as an untracked file to commit

### Requirement: Protobuf command interface
The firmware SHALL speak length-delimited protobuf over USB CDC serial,
generated from a single canonical `.proto` shared by firmware and host, decoded
on-device by a **no-std Rust protobuf codec** (static/`heapless` buffers, no
heap growth on the command path). Each command SHALL be a
`Command { uint32 id; oneof body { â€¦ } }` carrying an `id`; each reply SHALL be
a `Response { uint32 id; oneof body { â€¦ } }` echoing that `id` with an `ok` or
`err` variant. Framing SHALL be a fixed 2-byte little-endian length prefix with
NO application-layer CRC (USB bulk transfer guarantees integrity and in-order
delivery). The v1 command set is `info`, `ws2812`, `dshot`, `pwm`, `uart_tx`
(incl. SBUS-style frames), `i2c_scan`, `i2c_read`, `spi_tx`, `sync`, `loopback`
per docs/EXERCISER.md parameter shapes; binary fields (`uart_tx`/`spi_tx`
`payload`) SHALL be proto `bytes`, and the error reason SHALL be a typed
`ErrorReason` enum, not a free-form string.

#### Scenario: Response echoes id
- **WHEN** any valid command with `id = 42` is decoded and dispatched in the host-side codec/dispatch harness (no hardware)
- **THEN** the response message carries `id = 42` and an `ok` or `err` body variant

#### Scenario: Binary payload round-trips without corruption
- **WHEN** a `uart_tx`/`spi_tx` `payload` containing `0x00` and `0x22` bytes is encoded and decoded through the shared schema
- **THEN** the decoded bytes are byte-identical to the input, verified by a cross-language (Rust â†” Python) golden round-trip in CI

### Requirement: Host-testable policy/hardware separation
The policy layer SHALL be a host-buildable Rust module set â€” command decode,
dispatch, E9 policy, hazard checks, and timing â€” depending on neither the HAL
nor target hardware, so the full protocol/policy suite runs on the CI host.
HAL-bound code (PIO, IÂ˛C/SPI/PWM/UART/GPIO/ADC, USB) SHALL be isolated behind
that boundary.

#### Scenario: Policy suite runs with no target
- **WHEN** the policy/protocol test suite is run on the CI host
- **THEN** it builds and passes without a `thumbv8m` target toolchain, without the HAL crates, and with no hardware attached

## MODIFIED Requirements

### Requirement: Clock-parameterized timing
All timing constants SHALL derive from the configured system clock; migrating RP2040 (133 MHz) -> RP2350 (150 MHz) SHALL require only a build-configuration change (Cargo feature/target selection), no timing re-derivation.

#### Scenario: Two clocks, one source tree
- **WHEN** the firmware is built for the RP2040 (133 MHz) and RP2350 (150 MHz) configurations
- **THEN** both builds succeed from unmodified sources with PIO dividers and timing constants computed from the clock parameter, verified by host unit tests over both clock values

### Requirement: Headless CI build
The firmware SHALL build headless on Linux CI with a pinned Rust toolchain (open toolchain only): `cargo build` for the `thumbv8m.main-none-eabi` target is the automated gate. Compile success (plus the host-native policy suite) is the automated gate; real-silicon validation is explicitly a human step and remains an open gate until performed.

#### Scenario: CI compiles the firmware target
- **WHEN** CI runs the firmware job on a clean checkout
- **THEN** the `thumbv8m.main-none-eabi` firmware compiles and the host policy suite passes, using no account-walled or Windows-only tools, with hardware validation flagged as an unmet human gate
