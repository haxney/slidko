## Why

The exerciser firmware was written in C/pico-sdk on a rationale that turns out
not to hold: that the drone-critical protocols (DShot, WS2812, SBUS) had mature
C implementations with no Rust equivalent. They don't — those are PIO programs,
and PIO assembly ports to Rust verbatim (`pio-proc`), while `rp235x-hal` fully
supports the canonical RP2350A part (I²C/SPI/PWM/UART/GPIO/ADC). The only
genuinely "more mature in C" component is the USB **device stack** (TinyUSB vs
`embassy-usb`) — and the exerciser's use of it is the trivial corner: a single
CDC-ACM interface at trickle rate, the most-run path in embedded Rust.

The C firmware (`firmware/src/`, ~750 lines) is compile-verified but **never
validated on real silicon**, so it is disposable at no sunk cost. Getting it to
production would require hardware debug loops we cannot run right now (author
away from the bench) — and would then be thrown away for the Rust version
anyway. Moving to Rust now also lets the exerciser share a code substrate with
the deferred Rust pod control plane, collapsing a three-toolchain wire story
toward two. This is a deliberate reversal of the CLAUDE.md language ruling by
its owner, on the merits above — not a quiet drift.

Covers the docs/ROADMAP.md **"Parallel track — bench corpus & exerciser
firmware"** work item ("Exerciser firmware per EXERCISER.md"). No matching
items exist in docs/TASKS.md (the exerciser is tracked on the ROADMAP parallel
track, not itemized there).

## What Changes

- **BREAKING (toolchain):** replace the C/pico-sdk (CMake) firmware with a
  **Rust** implementation — `rp235x-hal` / `embassy-rp`, async CDC-ACM over
  `embassy-usb`, targeting `thumbv8m.main-none-eabi` on the RP2350A **ARM
  Cortex-M33** core (A4 stepping canonical). Flash via UF2/BOOTSEL or
  `picotool` (keeps CI headless; no dependence on probe-rs RP2350 debug).
- The v1 command set, the device-agnostic schema, and the `id`-echo /
  `ok|err` contract are **preserved exactly**: `info`, `ws2812`, `dshot`
  (150/300/600), `pwm`, `uart_tx` (incl. SBUS 8E2 inverted), `i2c_scan`,
  `i2c_read`, `spi_tx`, `sync`, `loopback`, with `adc_watch` reserved.
- DShot/WS2812 are generated by **inline PIO assembly** (`pio-proc`) carrying
  the identical cycle-exact timing constants from docs/EXERCISER.md; one PIO
  program serves all three DShot rates via clock divider (T1H = 2×T0H).
- The load-bearing **policy (no-HAL, host-testable) ↔ hw (HAL-bound)** split is
  preserved as Rust modules/crates; policy code (command decode, dispatch, E9
  policy, hazard, timing) unit-tests on the host with no target hardware.
- **BREAKING (wire):** JSON-lines is replaced by **length-delimited protobuf**,
  carrying over every wire decision from the superseded `exerciser-protobuf-wire`
  change (typed `oneof` envelope, `bytes` payloads, `ErrorReason` enum, fixed
  2-byte LE length prefix, no application-layer CRC over USB) — but the device
  codec is a **no-std Rust protobuf library** (candidates: `femtopb`,
  `micropb`, or `prost`+`heapless` — see design.md; this choice also carries a
  future-build-system cost recorded in `build-system-evaluation` design.md D6,
  see Impact) instead of nanopb(C).
- E9 A2/A4 stepping detection + silicon-aware self-limiting input policy,
  clock-parameterized timing (RP2040 dev @133 MHz ↔ RP2350 @150 MHz), and the
  full hazard envelope are **carried forward unchanged in behavior**, only
  re-expressed against the Rust HAL (stepping read via PAC/SYSINFO register).
- **Supersedes `exerciser-protobuf-wire`** (0/18 tasks, unimplemented): that
  change's nanopb-on-C plan is obsolete under a Rust firmware. Its wire
  contract survives here; the change itself should be archived as
  superseded-by this one (disposition noted in design.md).

### Modified Capabilities

- `exerciser-firmware`: the implementation substrate changes from C/pico-sdk to
  Rust (`rp235x-hal`/`embassy-rp`), the command interface from JSON-lines to
  length-delimited protobuf via a no-std Rust codec, and the toolchain/CI/SDK
  provisioning requirements from pico-sdk-vendoring to Cargo + pinned crates.
  The E9-guard, clock-parameterized-timing, and hazard-envelope requirements
  are preserved in behavior and re-expressed against the Rust HAL.

## Impact

- **Deleted:** the entire C tree under `firmware/src/` (`main.c`, `protocol/`,
  `hw/`), `firmware/CMakeLists.txt`, `pico_sdk_import.cmake`, the vendored
  `firmware/vendor/pico-sdk`, `firmware/test/` (C tests), and the
  `firmware/build-pico*/` trees. The `.pio` **timing content** is preserved by
  porting into `pio-proc` (the C `% c-sdk {}` init helpers are rewritten).
- **New build deps:** Rust stable + `thumbv8m.main-none-eabi` target,
  `rp235x-hal`/`embassy-rp`, `embassy-usb`, `pio`/`pio-proc`, a no-std protobuf
  crate, `picotool`/`elf2uf2` for UF2. All open, headless, CI-friendly,
  account-wall-free — clears the CLAUDE.md toolchain bar.
- **Docs updated:** CLAUDE.md "Language ruling" (the zero-Rust-in-v1 and
  C-for-exerciser rulings) and docs/EXERCISER.md "Platform / Toolchain" section
  (pico-sdk C → Rust). Guardrail 1 (no pod code) and guardrail 4 (no DUT
  write/control) remain in force — this is a language swap of the exerciser
  only, not a scope change.
- **Supersedes** `exerciser-protobuf-wire`; archive it as superseded.
- **Host side:** the greenfield Python host codec still generates from the same
  `.proto` — unaffected by the firmware language; only the device codec swaps.
- **Build-system coupling (codec choice → future Bazel cost):** the no-std codec
  decision has a downstream build-orchestration cost recorded in the
  `build-system-evaluation` change (design.md D6). `prost`+`heapless` gets a
  first-class Bazel proto rule (`rust_prost_library`) but needs `alloc` on the
  RP2350A; `micropb`/`femtopb` are alloc-free (better embedded fit) but have no
  Bazel proto rule, so their codegen becomes a hand-rolled `genrule` in any
  future Bazel world. This does **not** force the codec here — the current
  `mise`-based build shells out to whichever generator regardless — but the codec
  should be chosen in view of it (the open question is whether this firmware
  accepts `alloc`; if yes, `prost` keeps the cheaper-Bazel-later door open at low
  cost). No effect on this change's behavior or guardrails.
- **Hardware validation remains an open acceptance gate** that cannot be closed
  until the author is back at the bench (true for the C version equally).
