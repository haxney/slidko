## 4. Clock-parameterized timing tables

- [x] 4.1 Rewrote `test_timing.c` for real (the prior file computed nothing — `compute_dshot_timing`/`compute_ws2812_timing` were stubs returning a zeroed struct and every assertion was `assert(1==1)`): asserts `t1h_cycles == 2*t0h_cycles` for all 3 DShot rates at both 133 MHz and 150 MHz, that cycle counts differ per clock, that DShot600's bit period is shorter than DShot150's, and WS2812 T1H > T0H with both scaling up with clock
- [x] 4.2 Rewrote `timing.c`: `compute_dshot_timing`/`compute_ws2812_timing` now take `clk_hz` as an explicit parameter (not just the `SYS_CLK_HZ` macro) so one test binary can recompute both clocks without recompiling; fixed the previous version's broken unit math (it mixed a `bit_period_us` variable that was actually `us*100` with true-microsecond ratios, and rounded T0H/T1H independently which — verified by direct computation — breaks the "T1H is always 2xT0H" invariant at 133 MHz by 1 cycle; now `t1h_cycles = t0h_cycles * 2` by construction, satisfying the invariant exactly at any clock); native tests green
- [x] 4.3 Verified: both `pico` (RP2040, 133 MHz) and `pico2` (RP2350, 150 MHz) targets compile from the unmodified source tree — "two clocks, one source tree" confirmed for real, not just structurally. Getting RP2040 to actually build at 133 MHz surfaced a third real bug: `set_sys_clock_khz(133000, true)` doesn't work out of the box — `hardware/clocks.h` only auto-derives PLL parameters for its one canned 200 MHz recipe; any other `SYS_CLK_HZ` (including 133 MHz) needs `PLL_SYS_VCO_FREQ_HZ`/`PLL_SYS_POSTDIV1`/`PLL_SYS_POSTDIV2` supplied explicitly or the header `#error`s. Computed by hand: VCO = 12 MHz(XOSC) x FBDIV; since gcd(12,133)=1, FBDIV must itself be a multiple of 133, giving the unique in-range solution FBDIV=133 -> VCO=1596 MHz (within the 750-1600 MHz PLL range), POSTDIV1=6/POSTDIV2=2 (product 12, 1596/12=133 exactly). Also had to switch `SYS_CLK_HZ`/these three from `target_compile_definitions` (PRIVATE to our one target) to directory-scoped `add_compile_definitions()` set before `pico_sdk_init()`, since `hardware/clocks.h` needs them visible to every translation unit that includes it, not just our own sources
